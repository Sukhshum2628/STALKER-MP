# STALKER-MP Project Charter

Version: 0.1 (Draft)

---

# Project Mission

STALKER-MP exists to extend the X-Ray Engine so that multiple human players can inhabit the same persistent ALife simulation without sacrificing the systems that make S.T.A.L.K.E.R. unique.

This project is not a multiplayer game built on top of STALKER.

This project is the evolution of the existing simulation.

The Zone remains the protagonist.

Players become participants inside it.

---

# Project Vision

The ultimate vision of STALKER-MP is to create a world where:

- The Zone continues evolving even when nobody is connected.
- AI stalkers continue living their own lives.
- Mutants migrate naturally.
- Smart Terrains continue assigning jobs.
- Faction warfare continues progressing.
- Story progression remains consistent.
- Human players are simply additional stalkers inside that simulation.

The player should never feel like the universe revolves around them.

Instead, the player should feel like they entered an already-living world.

---

# Engineering Philosophy

Every architectural decision made during this project must answer one question:

"Does this preserve the feeling that the Zone is alive?"

If the answer is "No",

the implementation should be reconsidered.

---

# Design Principles

## Principle 1 — Preserve Before Replace

Existing engine systems represent years of development and debugging.

The preferred order is:

1. Reuse existing systems.
2. Extend existing systems.
3. Abstract existing systems.
4. Replace only when absolutely necessary.

Replacing mature engine systems is considered the last resort.

---

## Principle 2 — The Zone Is Authoritative

The authoritative simulation belongs to the Zone.

The Host merely executes it.

Players do not own the simulation.

Players influence the simulation.

---

## Principle 3 — One World

There is exactly one world.

There are never multiple copies of the Zone.

Every player observes and interacts with the same simulation.

---

## Principle 4 — Multiple Human Stalkers

Human players are not special entities.

Conceptually,

they are stalkers whose decisions originate from human input instead of the GOAP planner.

The engine should treat them as naturally as possible.

---

## Principle 5 — Preserve ALife

ALife is the defining feature of the X-Ray Engine.

The objective is never to replace ALife.

The objective is to allow ALife to support multiple human participants.

---

## Principle 6 — Host Authority

Only the Host may authoritatively execute:

- ALife
- AI
- Quest progression
- Inventory ownership
- Physics authority
- Save-game persistence
- Story progression
- Faction simulation
- Smart Terrain allocation

Clients never become authoritative over world state.

---

## Principle 7 — Predict Locally, Simulate Centrally

Clients may predict:

- movement
- animations
- camera
- local effects

Only the Host simulates:

- AI
- NPC decisions
- combat outcomes
- inventory changes
- persistent world state

---

## Principle 8 — Compatibility Matters

Engine modifications should preserve compatibility with:

- Anomaly
- GAMMA
- ZONA

whenever technically practical.

Avoid unnecessary engine rewrites.

---

# Project Invariants

The following rules must never be violated.

## Invariant 1

There is exactly one authoritative ALife simulation.

---

## Invariant 2

Every entity has exactly one authoritative owner.

---

## Invariant 3

Every persistent object exists only once.

---

## Invariant 4

Players never create independent simulations.

---

## Invariant 5

AI and human players share the same world state.

---

## Invariant 6

Offline simulation continues regardless of player locations.

---

## Invariant 7

Networking must never replace ALife.

Networking only transports state generated by ALife.

---

## Invariant 8

The simulation never pauses because players disconnect.

# What We Are Building

We are building:

✓ Persistent multiplayer framework

✓ Shared ALife simulation

✓ Host authoritative architecture

✓ Multiple simultaneous human stalkers

✓ Persistent progression

✓ Long-running worlds

✓ Future dedicated server support

# What We Are NOT Building

We are NOT building:

✗ Deathmatch

✗ Team Deathmatch

✗ Battle Royale

✗ MMO

✗ Client-authoritative networking

✗ Peer-to-peer simulation

✗ Complete engine rewrite

✗ Replacement AI

✗ Replacement ALife

---

## Objective 4

---

## Objective 5

Preserve existing scripting interfaces whenever possible.

---

## Objective 6

---

# Engineering Rules

Before modifying any subsystem ask:

Can this be reused?

Can this be abstracted?

Can this remain untouched?

Only if all answers are "No"

should replacement be considered.

---

# Development Methodology

Every implementation follows:

Research

↓

Architecture Decision

↓

RFC

↓

Sprint Specification

↓

Implementation

↓

Validation

↓

Review

↓

Merge

No implementation begins without a corresponding RFC.

---

# Sprint Philosophy

Each sprint has exactly one engineering objective.

Large features must be decomposed into independently testable milestones.

Every sprint must leave the engine in a buildable state.

---

# Success Criteria

The project is considered successful when:

• Multiple human players inhabit one persistent Zone.

• ALife continues functioning naturally.

• Existing gameplay systems remain intact.

• The simulation remains authoritative.

• Performance scales predictably.

• The project remains maintainable.

---

# Final Statement

STALKER-MP is not an attempt to make STALKER behave like other multiplayer games.

It is an attempt to allow multiple players to exist inside the same autonomous simulation while preserving the identity of the X-Ray Engine.

Every future architectural decision should reinforce that goal.
